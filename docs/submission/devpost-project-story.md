## Inspiration

Reduced-motion support is often implemented as a blanket `animation: none`. That can reduce discomfort, but it can also erase information. A progress shimmer communicates that work is still happening. A success animation confirms an outcome. A sliding panel explains hierarchy and where focus moved.

Accessibility guidance reflects this tension. WCAG 2.3.3 requires interaction-triggered motion to be disableable unless it is essential to functionality or information, while Apple’s Reduced Motion criteria explicitly warns against removing animation entirely when it communicates status or hierarchy. We built StillMeaning around one principle: **reduce motion, not meaning**.

## What it does

StillMeaning is an AI-assisted motion accessibility workbench for frontend developers, design-system teams, accessibility engineers, and product designers.

The demo includes three stable examples: upload progress, save confirmation, and panel hierarchy. For each animation, StillMeaning identifies the technique, semantic role, motion risk, and supporting evidence. It then proposes a safer implementation, explains what meaning must remain, and shows the exact code diff.

The central experience is a side-by-side Original versus StillMeaning Version comparison. Reviewers can switch between normal and reduced-motion modes, inspect a bounded Meaning Preserved checklist, and copy the generated code. Developers can also paste HTML, CSS, JavaScript, or React source for analysis when live GPT-5.6 access is available.

## How we built it

StillMeaning is a Next.js and React application written in strict TypeScript. A server-only analysis route validates requests with Zod, accepts either a known example or bounded source text, and calls the OpenAI Responses API. Model output uses strict structured output and is parsed again before the UI can consume it.

The three demo previews are hand-authored components. Generated or pasted code is never executed. The workbench renders model output as text, preserves the last valid result when analysis fails, and exposes provenance as either `Live · GPT-5.6` or `Demo fallback`.

The fallback is intentionally narrow: it supports only the three curated examples and is visibly disclosed. StillMeaning never pretends that deterministic fixture data came from GPT-5.6, and it never fabricates an analysis for arbitrary pasted source.

## How GPT-5.6 is integrated

GPT-5.6 performs the core reasoning task rather than acting as a chat decoration. Given untrusted frontend source or a curated example, it must:

1. identify the animation technique and original behavior;
2. infer the semantic role carried by the motion;
3. assess motion risk and explain the evidence;
4. select a lower-motion or non-motion strategy;
5. generate usable replacement code;
6. state which meaning is preserved; and
7. return machine-checkable validation evidence.

The provider uses the Responses API with `store: false`, a 30-second service timeout, a strict Zod-backed schema, bounded input and output, and application-side re-validation. The API key stays on the server. The model cannot set its own provenance, and no generated code is executed in the product.

## Challenges

The hardest problem was defining an honest validation boundary. Source code alone cannot prove that every user will understand an alternative or that an interface is WCAG-conformant. We therefore validate bounded evidence—such as retained status text, ARIA state, focus destination, and context—without claiming certification.

We also needed a demo that remains dependable when a model or network is unavailable. The solution was a visibly labeled fallback for known examples only, while arbitrary pasted source fails honestly and preserves the previous valid result.

Finally, showing generated UI code creates a security temptation: executing it would make the demo more dynamic but would introduce an unnecessary untrusted-code boundary. We chose inspectable text diffs and hand-authored previews for the MVP.

## What we learned

Motion accessibility is not a binary animation-on or animation-off decision. The useful unit of analysis is the meaning carried by the animation: status, progress, hierarchy, focus, feedback, or causality. Once that role is explicit, safer alternatives become much more concrete.

We also learned that AI remediation is most trustworthy when the product exposes reasoning, provenance, constraints, and code changes together. A generated snippet alone is not enough for developer review.

## What’s next

The next version would add AST-based animation extraction, repository and design-system context, a hardened sandbox for opt-in preview compilation, pull-request review comments, Storybook integration, and a larger evaluated corpus of motion patterns. The goal is not to become another broad accessibility scanner; it is to become the semantic transformation layer for motion accessibility.
